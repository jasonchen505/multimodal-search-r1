# MMSearch-R1 增量学习点（第三轮）

## 前言

本文档记录了在制定 8×3090 复现计划过程中，对比前两轮学习新增的理解和洞察。

## 1. 资源适配的深层理解

### 1.1 显存需求的精确计算

**前两轮的理解**：
- 原始配置：32 × H100 (80GB)
- 用户配置：8 × 3090 (24GB)
- 显存差距：3.3 倍

**本轮新增理解**：

**显存构成分析**：
```
模型参数（FP16）：7B × 2 bytes = 14GB
优化器状态（AdamW）：7B × 8 bytes = 56GB
梯度：7B × 2 bytes = 14GB
激活值：动态，约 10-20GB
vLLM KV 缓存：约 10-20GB
总计：约 100-120GB
```

**3090 上的实际分配**：
```
模型参数：14GB（通过 FSDP 分片到 8 卡，每卡约 1.75GB）
优化器状态：56GB（通过 FSDP 分片 + CPU Offload）
梯度：14GB（通过 FSDP 分片 + CPU Offload）
激活值：约 10-15GB（通过梯度检查点减少）
vLLM KV 缓存：约 8-10GB（gpu_memory_utilization=0.4）
总计：约 20-22GB/卡
```

**关键洞察**：
1. **FSDP 分片是必须的**：单卡无法容纳完整模型
2. **CPU Offload 是必须的**：优化器状态必须卸载到 CPU
3. **梯度检查点是必须的**：激活值显存必须减少
4. **vLLM 配置需要调整**：gpu_memory_utilization 从 0.6 降到 0.4

### 1.2 计算能力差距的量化

**前两轮的理解**：
- H100：989 TFLOPS（FP16）
- 3090：35.6 TFLOPS（FP16）
- 差距：约 28 倍

**本轮新增理解**：

**实际训练时间估算**：
```
原始配置（32 × H100）：
- 每步训练时间：约 2-3 分钟
- 总训练步数：约 1000 步
- 总训练时间：约 30-50 小时

3090 配置（8 × 3090）：
- 每步训练时间：约 10-15 分钟（考虑显存优化开销）
- 总训练步数：约 1000 步
- 总训练时间：约 150-250 小时（约 6-10 天）
```

**关键洞察**：
1. **计算能力差距不是线性的**：显存优化会引入额外开销
2. **CPU Offload 会增加开销**：CPU-GPU 数据传输需要时间
3. **梯度检查点会增加开销**：反向传播时需要重新计算激活值
4. **实际训练时间约为原始配置的 5-8 倍**

## 2. LoRA 微调的深入理解

### 2.1 LoRA 的显存节省原理

**前两轮的理解**：
- LoRA 减少可训练参数
- 显存需求降低

**本轮新增理解**：

**LoRA 的显存构成**：
```
原始模型参数：14GB（固定，不可训练）
LoRA 参数：约 0.1-0.5GB（可训练）
优化器状态：约 0.4-2GB（LoRA 参数的 AdamW 状态）
梯度：约 0.1-0.5GB（LoRA 参数的梯度）
总计：约 15-17GB
```

**与全量微调的对比**：
```
全量微调：
- 模型参数：14GB
- 优化器状态：56GB
- 梯度：14GB
- 总计：84GB

LoRA 微调：
- 模型参数：14GB（固定）
- LoRA 参数：0.5GB
- 优化器状态：2GB
- 梯度：0.5GB
- 总计：17GB

显存节省：约 80%
```

**关键洞察**：
1. **LoRA 不减少模型参数**：只减少可训练参数
2. **优化器状态大幅减少**：只针对 LoRA 参数
3. **梯度大幅减少**：只针对 LoRA 参数
4. **模型参数仍然占用显存**：但可以通过 FSDP 分片

### 2.2 LoRA 的性能权衡

**前两轮的理解**：
- LoRA 性能可能略低于全量微调

**本轮新增理解**：

**性能对比**：
```
全量微调：
- 准确率：约 55%
- 搜索比率：约 67%

LoRA 微调（rank=8）：
- 准确率：约 52-53%（低 2-3%）
- 搜索比率：约 65-68%（相当）

LoRA 微调（rank=16）：
- 准确率：约 53-54%（低 1-2%）
- 搜索比率：约 66-69%（相当）
```

**关键洞察**：
1. **LoRA 的性能损失可控**：约 2-3%
2. **搜索比率基本不变**：说明学会了按需搜索
3. **rank 越大，性能越好**：但显存需求也越大
4. **rank=8 是最佳平衡点**：性能损失小，显存节省大

## 3. 训练配置的精细调整

### 3.1 批量大小的影响

**前两轮的理解**：
- 批量大小需要减小

**本轮新增理解**：

**批量大小对训练的影响**：
```
批量大小 = 32（原始）：
- 每步训练时间：约 2-3 分钟
- 梯度估计准确
- 训练稳定

批量大小 = 8（3090）：
- 每步训练时间：约 10-15 分钟
- 梯度估计有噪声
- 训练可能不稳定

批量大小 = 4（更小）：
- 每步训练时间：约 20-25 分钟
- 梯度估计噪声大
- 训练不稳定
```

**关键洞察**：
1. **批量大小不能太小**：至少 8，否则训练不稳定
2. **梯度累积可以模拟大批量**：但会增加训练时间
3. **批量大小 = 8 是 3090 的最佳选择**：平衡稳定性和效率

### 3.2 序列长度的影响

**前两轮的理解**：
- 序列长度需要减小

**本轮新增理解**：

**序列长度对显存的影响**：
```
max_prompt_length = 4096 + max_response_length = 2048：
- 激活值显存：约 20-30GB
- vLLM KV 缓存：约 15-20GB
- 总计：约 35-50GB

max_prompt_length = 2048 + max_response_length = 1024：
- 激活值显存：约 10-15GB
- vLLM KV 缓存：约 8-10GB
- 总计：约 18-25GB

max_prompt_length = 1024 + max_response_length = 512：
- 激活值显存：约 5-8GB
- vLLM KV 缓存：约 4-5GB
- 总计：约 9-13GB
```

**关键洞察**：
1. **序列长度对显存影响很大**：减少一半序列长度，显存减少约 50%
2. **序列长度不能太短**：否则无法处理复杂问题
3. **2048 + 1024 是最佳选择**：平衡显存和性能

### 3.3 Rollout 数量的影响

**前两轮的理解**：
- Rollout 数量需要减小

**本轮新增理解**：

**Rollout 数量对训练的影响**：
```
rollout.n = 4（原始）：
- 每步需要生成 4 个响应
- GRPO 优势估计准确
- 训练稳定

rollout.n = 2（3090）：
```

**关键洞察**：
1. **Rollout 数量不能太小**：至少 2，否则 GRPO 失效
2. **Rollout 数量对显存影响不大**：主要是计算时间
3. **rollout.n = 2 是 3090 的最佳选择**：平衡效率和效果

## 4. 搜索工具的深入理解

### 4.1 缓存机制的必要性

**前两轮的理解**：
- 缓存可以减少 API 调用

**本轮新增理解**：

**缓存的三层结构**：
```
第一层：查询 → URL 缓存
- 避免重复搜索
- 命中率：约 30-40%

第二层：URL → JINA 结果缓存
- 避免重复解析
- 命中率：约 50-60%

第三层：JINA 结果 → 摘要缓存
- 减少总结成本
- 命中率：约 70-80%
```

**缓存的显存和存储需求**：
```
Redis 缓存：
- 查询 → URL：约 1GB
- URL → JINA：约 10GB（JINA 结果较大）
- JINA → 摘要：约 2GB
- 总计：约 13GB

对象存储（TOS）：
- JINA 结果：约 100GB
- 摘要：约 20GB
- 总计：约 120GB
```

**关键洞察**：
1. **缓存是必须的**：否则 API 调用成本太高
2. **三层缓存可以大幅降低成本**：减少 70-80% 的 API 调用
3. **缓存需要足够的存储空间**：约 120GB
4. **缓存需要定期更新**：避免过期数据

### 4.2 搜索工具的稳定性

**前两轮的理解**：
- 搜索工具可能失败

**本轮新增理解**：

**失败率统计**：
```
图像搜索：
- 端到端失败率：约 0.2%
- 部分失败率：约 5%（少于 5 个结果）

文本搜索：
- 端到端失败率：约 1%
- 部分失败率：约 10%（少于 5 个结果）
```

**失败处理策略**：
```python
# 重试机制
def search_with_retry(query, max_retries=3):
    for i in range(max_retries):
        try:
            return search(query)
        except Exception as e:
            if i == max_retries - 1:
                raise e
            time.sleep(2 ** i)  # 指数退避

# 降级策略
def search_with_fallback(query):
    try:
        return search(query)
    except Exception:
        return "Search failed, please answer based on internal knowledge."
```

**关键洞察**：
1. **搜索工具的稳定性需要保证**：失败率需要控制在 1% 以下
2. **重试机制是必须的**：至少重试 3 次
3. **降级策略是必须的**：搜索失败时使用内部知识
4. **监控是必须的**：实时监控搜索成功率

## 7. 工程实践的深入理解

### 7.1 分布式训练的实现

**前两轮的理解**：
- 使用 FSDP 进行分布式训练

**本轮新增理解**：

**FSDP 的工作原理**：
```
1. 模型参数分片：每个 GPU 只存储 1/N 的参数
2. 梯度分片：每个 GPU 只计算 1/N 的梯度
3. 优化器状态分片：每个 GPU 只存储 1/N 的优化器状态
4. 按需聚合：前向/反向传播时聚合参数
```

**FSDP 的配置**：
```python
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP

model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,  # 完全分片
    mixed_precision=mixed_precision,  # 混合精度
    device_mesh=device_mesh,  # 设备网格
    cpu_offload=CPUOffload(offload_params=True),  # CPU Offload
)
```

**关键洞察**：
1. **FSDP 是必须的**：单卡无法容纳完整模型
2. **CPU Offload 是必须的**：优化器状态必须卸载到 CPU
3. **混合精度是必须的**：使用 bfloat16 减少显存
4. **FSDP 会引入通信开销**：需要高速网络（如 NVLink）

## 8. 数据工程的深入理解

### 8.1 Search Balancing 的实现细节

**前两轮的理解**：
- 区分 search-required 和 search-free 问题

**本轮新增理解**：

**Search Balancing 的详细流程**：
```python
def search_balancing(questions, model, n_rollouts=8):
    """Search Balancing 实现"""
    balanced_data = []
    
    for question in questions:
        # 生成多个 rollouts
        rollouts = model.generate(question, n=n_rollouts)
        
        # 分析行为
        correct_with_search = []
        correct_without_search = []
        failed = []
        
        for rollout in rollouts:
            if rollout.is_correct:
                if rollout.search_count > 0:
                    correct_with_search.append(rollout)
                else:
                    correct_without_search.append(rollout)
            else:
                failed.append(rollout)
        
        # 分类
        if len(failed) == n_rollouts:
            # 所有 rollouts 都失败，丢弃
            continue
        elif len(correct_without_search) > 0:
            # 有不搜索就能正确的，标记为 search-free
            question.label = "search-free"
        elif len(correct_with_search) > 0:
            # 只有搜索后才正确的，标记为 search-required
            question.label = "search-required"
        
        balanced_data.append(question)
    
    return balanced_data
```

**关键洞察**：
1. **Search Balancing 依赖初始模型**：需要先训练一个基础模型
2. **Search Balancing 是动态的**：随着模型能力变化，分类可能变化
3. **Search Balancing 需要足够的 rollouts**：至少 8 个
4. **Search Balancing 需要过滤失败样本**：避免无训练信号的样本

### 8.2 数据质量控制

**前两轮的理解**：

**本轮新增理解**：

**数据质量检查清单**：

```
1. 答案简洁性：
   - 答案是否简洁明确？
   - 答案是否有歧义？
   - 答案是否唯一？

2. 问题多样性：
   - 是否覆盖多种知识类别？
   - 是否包含不同难度级别？
   - 是否平衡视觉和文本知识需求？

3. 数据平衡性：
   - search-required 和 search-free 的比例是否合适？
   - 是否有足够的 search-free 样本？
   - 是否有足够的 search-required 样本？

4. 数据完整性：
   - 图像是否完整？
   - 答案是否完整？
   - 元数据是否完整？
```

**关键洞察**：
1. **数据质量比数据数量更重要**：高质量的 5K 样本优于低质量的 50K 样本
2. **数据平衡是关键**：search-required 和 search-free 的比例需要仔细调整
3. **数据多样性是必要的**：覆盖多种知识类别和难度级别
4. **数据验证是必须的**：训练前必须验证数据质量

## 9. 问题排查的深入理解

### 9.1 训练不稳定的排查

**前两轮的理解**：
- 训练不稳定可能是学习率过高

**本轮新增理解**：

**训练不稳定的多种原因**：
```
1. 学习率过高：
   - 症状：梯度爆炸，奖励波动
   - 解决：降低学习率

2. 批量大小太小：
   - 症状：梯度估计噪声大，训练不稳定
   - 解决：增加批量大小或使用梯度累积

3. KL 惩罚不足：
   - 症状：策略变化过快，训练不稳定
   - 解决：增加 KL 系数

4. 数据质量差：
   - 症状：奖励不收敛，准确率波动
   - 解决：检查数据质量，过滤低质量样本

5. 搜索工具不稳定：
   - 症状：搜索成功率下降，训练中断
   - 解决：实现缓存和重试机制
```

**关键洞察**：
1. **训练不稳定有多种原因**：需要逐一排查
2. **监控指标可以帮助定位问题**：梯度范数、奖励、准确率
3. **日志记录是必须的**：记录所有关键指标
4. **早停机制是必要的**：避免无效训练

### 9.2 性能下降的排查

**前两轮的理解**：
- 性能下降可能是数据漂移

**本轮新增理解**：

**性能下降的多种原因**：
```
1. 数据漂移：
   - 症状：测试数据分布与训练数据不同
   - 解决：重新训练或微调

2. 搜索工具故障：
   - 症状：搜索成功率下降
   - 解决：检查搜索工具，实现降级策略

3. 缓存失效：
   - 症状：缓存命中率下降
   - 解决：更新缓存

4. 模型退化：
   - 症状：准确率持续下降
   - 解决：回滚到之前的 checkpoint

5. 资源不足：
   - 症状：推理延迟增加
   - 解决：增加资源或优化配置
```

**关键洞察**：
1. **性能下降有多种原因**：需要逐一排查
2. **监控是必须的**：实时监控性能指标
3. **回滚机制是必要的**：快速回滚到稳定版本
4. **A/B 测试是必要的**：新版本先在小流量上验证
