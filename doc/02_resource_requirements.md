# MMSearch-R1 计算资源需求评估

## 原始训练配置

### 硬件配置
- **GPU**: 32 × H100（80GB 显存）
- **节点**: 4 个节点，每个节点 8 卡
- **内存**: 每节点约 500GB+ 系统内存
- **存储**: 高速 NVMe SSD，用于数据加载和检查点

### 模型配置
- **基础模型**: Qwen2.5-VL-7B-Instruct（约 7B 参数）
- **模型大小**: 约 14GB（FP16）或 7GB（INT8）
- **优化器状态**: 约 28GB（AdamW）
- **梯度**: 约 14GB
- **激活值**: 动态，取决于序列长度

### 训练配置
- **批量大小**: 32（全局）
- **每 GPU 批量**: 4（32 / 8）
- **序列长度**: 4096（提示）+ 2048（响应）= 6144 tokens
- **总响应长度**: 8192 tokens（多轮）
- **Rollout 数量**: 4（每个提示生成 4 个响应）

## 显存需求分析

### 模型显存需求
```
模型参数（FP16）: 7B × 2 bytes = 14GB
优化器状态（AdamW）: 7B × 4 bytes × 2 = 56GB
梯度: 7B × 2 bytes = 14GB
总计: ~84GB（仅模型和优化器）
```

### 激活值显存需求
```
序列长度: 6144 tokens
批量大小: 4（每 GPU）
隐藏维度: 3584（Qwen2.5-VL-7B）
层数: 28
注意力头数: 28

激活值估算: ~10-20GB（取决于实现）
```

### vLLM 推理显存需求
```
KV 缓存: 取决于序列长度和批量大小
模型权重: 需要额外副本用于推理
总计: ~20-30GB
```

### 总显存需求
```
原始配置（H100 80GB）: ~120-150GB / GPU
实际使用: 约 60-70GB / GPU（通过 FSDP 分片）
```

## 用户资源对比

### 用户硬件
- **8 卡 3090**: 24GB 显存 / 卡
- **8 卡 4090**: 24GB 显存 / 卡

### 显存差距分析
```
H100（80GB） vs 3090/4090（24GB）: 3.3 倍差距
原始配置需要: ~60-70GB / GPU
用户可用: 24GB / GPU
差距: 约 2.5-3 倍
```

## 挑战与限制

### 1. 显存限制
- 无法使用原始批量大小
- 需要启用梯度检查点
- 需要启用参数/优化器卸载
- 可能需要使用 LoRA 或 QLoRA

### 2. 计算限制
- 3090/4090 的计算能力低于 H100
- 训练时间会显著增加
- 可能需要减少 rollout 数量

### 3. 通信限制
- 多节点训练需要高速网络
- 8 卡配置可能需要调整并行策略

## 优化策略

### 1. 显存优化
- **梯度检查点**: 减少激活值显存占用
- **参数卸载**: 将优化器状态卸载到 CPU
- **混合精度训练**: 使用 BF16/FP16
- **FSDP**: 完全分片数据并行

### 2. 计算优化
- **减少批量大小**: 从 32 减少到 8 或 16
- **减少 rollout 数量**: 从 4 减少到 2
- **减少序列长度**: 缩短最大响应长度
- **使用 LoRA**: 减少可训练参数数量

### 3. 推理优化
- **调整 vLLM 配置**: 减少 GPU 内存利用率
- **减少并发请求数**: 降低推理显存压力
- **使用更小的模型**: 考虑使用 3B 或 1.5B 模型

## 结论

### 完全复现的挑战
- 原始配置需要 32 × H100（80GB），用户只有 8 × 3090/4090（24GB）
- 显存差距约 3.3 倍，计算能力差距约 2-3 倍
- 完全复现原始配置几乎不可能

### 可行的复现方案
1. **使用 LoRA/QLoRA**: 减少可训练参数，降低显存需求
2. **减少批量大小**: 从 32 减少到 8 或 16
3. **减少 rollout 数量**: 从 4 减少到 2
4. **启用显存优化**: 梯度检查点、参数卸载等
5. **使用更小的模型**: 考虑 3B 或 1.5B 版本

### 预期效果
- 训练时间会增加 3-5 倍
- 可能需要更多实验来调整超参数
- 最终性能可能略有下降
