## 2. 消融实验数据

### 2.1 奖励设计消融

**表 2：EM vs GPT-4o 奖励对比**

| 奖励类型 | 平均准确率 | 平均搜索比率 |
|----------|-----------|-------------|
| EM（精确匹配） | 55.7% | 77.3% |
| GPT-4o（语义匹配） | 59.5% | 82.6% |

**分析**：
- GPT-4o 奖励提升 3.8%
- 但搜索比率也增加 5.3%
- EM 奖励更保守，搜索更精准

### 2.2 数据平衡消融

**表 3：数据平衡的影响**

| 配置 | 奖励 | 搜索比率 | 说明 |
|------|------|----------|------|
| 完整模型 | 相当 | 低且稳定 | 平衡数据 + 搜索惩罚 |
| 无搜索惩罚 | 略高 | 快速收敛到 100% | 模型过度使用搜索 |
| 无数据平衡 | 略高 | 快速收敛到 100% | 缺少 search-free 样本 |

**核心发现**：
> "Removing either the search penalty or data balancing leads to distinct trade-offs. Although both ablated variants achieve slightly higher rewards, they do so at the cost of overusing the search tool."

### 2.3 RL vs SFT 对比

**表 4：RL 与 SFT 的性能对比**

| 方法 | 数据量 | FVQA-test | InfoSeek | MMSearch | SimpleVQA | LiveVQA |
|------|--------|-----------|----------|----------|-----------|---------|
| SFT | 8,000 | +2.1% | +1.8% | +3.2% | +2.5% | +1.9% |
| RL | 5,000 | +3.1% | +2.8% | +4.2% | +3.5% | +2.4% |

**分析**：
- RL 使用更少数据（5K vs 8K）
- RL 性能提升更大（平均 +3.2% vs +2.3%）
- RL 行为更符合任务需求

## 3. 训练动态分析

### 3.1 奖励收敛曲线

**典型训练动态**：
- **Step 0-10**：奖励快速上升，模型学习基本格式
- **Step 10-30**：奖励稳定增长，模型学习搜索策略
- **Step 30-50**：奖励收敛，搜索比率稳定

**关键观察**：
- 搜索比率在 Step 20 左右达到峰值
- 之后逐渐下降，模型学会更精准的搜索
- 最终收敛到 60-70% 的搜索比率

### 3.2 搜索行为演变

**阶段 1：探索期（Step 0-10）**
- 搜索比率快速上升
- 模型尝试各种搜索策略
- 奖励波动较大

**阶段 2：学习期（Step 10-30）**
- 搜索比率开始下降
- 模型学会区分 search-required 和 search-free
- 奖励稳定增长

**阶段 3：收敛期（Step 30+）**
- 搜索比率稳定
- 奖励趋于平稳
- 模型行为成熟

## 4. 实践注意事项

### 4.1 训练稳定性问题

**常见问题**：
1. **梯度爆炸**：`grad_norms → inf`
   - 原因：批量大小配置不当
   - 解决：正确配置 `train_batch_size` 和 `ppo_mini_batch_size`

2. **搜索比率过高**：模型过度使用搜索
   - 原因：缺少搜索惩罚或数据不平衡
   - 解决：添加搜索惩罚，确保数据平衡

3. **奖励不收敛**：训练不稳定
   - 原因：学习率过高或 KL 惩罚不足
   - 解决：降低学习率，增加 KL 系数

### 4.2 超参数调优建议

**学习率**：
- 推荐：2e-6
- 范围：1e-6 到 5e-6
- 调优：如果训练不稳定，降低学习率

**KL 系数**：
- 推荐：0.001
- 范围：0.0005 到 0.005
- 调优：如果策略变化过快，增加 KL 系数

**搜索惩罚**：
- 推荐：0.1
- 范围：0.05 到 0.2
- 调优：如果搜索比率过高，增加惩罚

**格式惩罚**：
- 推荐：0.1
- 范围：0.05 到 0.2
- 调优：如果格式错误较多，增加惩罚

### 4.3 数据准备建议

**数据平衡**：
- Search-required：约 70%
- Search-free：约 30%
- 确保有足够的 search-free 样本

**数据质量**：
- 答案必须简洁明确
- 问题必须有唯一正确答案
- 避免歧义性问题

**数据多样性**：
- 覆盖多种知识类别
- 包含不同难度级别
- 平衡视觉和文本知识需求

### 4.4 搜索工具配置

**图像搜索**：
- 使用 SerpAPI
- 返回 top-5 结果
- 实现缓存避免重复调用

**文本搜索**：
- SerpAPI → JINA Reader → Qwen3-32B
- 返回 top-5 摘要
- 实现三层缓存

**缓存策略**：
- 查询 → URL 缓存
- URL → JINA 结果缓存
- JINA 结果 → 摘要缓存

## 5. 性能优化技巧

### 5.1 显存优化

**梯度检查点**：
```python
model.gradient_checkpointing_enable()
```

**参数卸载**：
```python
fsdp_config.param_offload = True
fsdp_config.optimizer_offload = True
```

**混合精度**：
```python
torch_dtype = torch.bfloat16
```

### 5.2 计算优化

**批量大小调整**：
- 根据显存调整 `train_batch_size`
- 使用梯度累积模拟大批量

**序列长度优化**：
- 减少 `max_prompt_length`
- 减少 `max_response_length`
- 减少 `response_length_total`

**Rollout 优化**：
- 减少 `rollout.n`（从 8 降到 4 或 2）
- 减少 `max_gen_round`（从 3 降到 2）

### 5.3 推理优化

**vLLM 配置**：
```yaml
gpu_memory_utilization: 0.4  # 降低以节省显存
tensor_model_parallel_size: 1  # 单卡推理
enable_chunked_prefill: False
enforce_eager: False
```

## 6. 评估与调试

### 6.1 训练监控指标

**关键指标**：
1. **奖励（Reward）**：应稳步上升
2. **搜索比率（Search Ratio）**：应先升后降，最终稳定
3. **准确率（Accuracy）**：应稳步上升
4. **梯度范数（Grad Norm）**：应保持稳定，不应爆炸

**监控工具**：
- WandB：记录训练指标
- TensorBoard：可视化训练过程
- 自定义日志：记录详细信息

### 6.2 验证策略

**验证频率**：
- 每 100 步验证一次
- 记录验证集上的准确率和搜索比率

**验证指标**：
- 准确率（LLM-as-Judge）
- 搜索比率
- 格式正确率

### 6.3 常见问题诊断

**问题 1：奖励不收敛**
- 检查学习率是否过高
- 检查 KL 系数是否足够
- 检查数据质量

**问题 2：搜索比率过高**
- 检查搜索惩罚是否启用
- 检查数据是否平衡
- 检查 search-free 样本是否足够

**问题 3：格式错误率高**
- 检查格式惩罚是否启用
- 检查提示模板是否正确
- 检查模型是否理解格式要求

## 总结

MMSearch-R1 的成功源于：

1. **精心设计的数据**：搜索平衡的数据集
2. **合理的奖励设计**：准确性 + 搜索惩罚 + 格式奖励
3. **端到端 RL 训练**：直接优化搜索行为
4. **工程优化**：多层缓存、分布式限速

通过理解这些关键因素，我们可以更好地复现和扩展该项目。
